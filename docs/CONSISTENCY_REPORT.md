# Consistency of the Discovery Service Under Horizontal Scaling

**Author:** Ermiyas (GitHub: ermi9)
**Date:** 2026-05-08
**Project:** microservice-discovery-k8s

---

## Introduction

Service discovery is a foundational concern in any microservice architecture: clients must be able to locate healthy service instances without hard-coded network addresses, and the registry that mediates this lookup must remain correct even as the discovery layer itself is scaled out. This report examines the consistency properties of a Spring Boot discovery service deployed on Kubernetes, focusing on the changes required to make it safe to run as three concurrent replicas backed by a Redis master/replica cluster. The original implementation contained both compile-time defects and a subtler runtime hazard: a periodic health-check task that, when executed concurrently on multiple replicas, produced non-deterministic failure counts and therefore inconsistent service state. Resolving these issues required code corrections, a Redis-backed distributed lock, and an updated Deployment manifest.

## Architecture Overview

The system is composed of three logical tiers. The **discovery-service** is a Spring Boot application that exposes REST endpoints for service registration, deregistration, and lookup, and runs a `@Scheduled` health-check job. State is externalised to **Redis**, deployed as a master/replica cluster: a single `redis-master` Deployment with append-only-file (AOF) persistence on a 1 GiB PersistentVolumeClaim, and a `redis-replica` Deployment with two replicas that asynchronously replicate from the master. The Spring Data Redis layer is configured through Lettuce using `RedisStaticMasterReplicaConfiguration` with a `REPLICA_PREFERRED` read-from policy, so writes are routed to the master while reads are served preferentially from the replicas. Custom converters serialise `LocalDateTime` values to and from `byte[]` for Redis hash storage. Two domain entities sit in this store: `Service`, keyed by `name` as the Redis `@Id`, and `PodSnapshot`, keyed by an `@Indexed` `serviceName` field used by `PodSnapshotRepository.findByServiceName()`. The discovery-service Deployment was originally configured with `replicas: 1`; this report concerns the move to `replicas: 3`.

## Consistency Challenges at Scale

Running the discovery-service as a single instance hides several latent correctness problems that surface immediately under horizontal scaling.

The first class of problems is structural. `Service.java` declared `getId()` and `setId(UUID)` accessors that referenced an `id` field that did not exist on the class, and the file did not import `java.util.UUID`. These were compile-time errors, but they also reflected a conceptual inconsistency: the Redis identity of a `Service` is its `name` (a `String`), not a synthetic UUID. Leaving the unused accessors in place would risk a caller writing through a "ghost" identifier that the persistence layer ignores, producing entries that appear to be saved but cannot be retrieved by primary key. The same shape of bug existed in `PodSnapshot.java`, where the constructor assigned a non-existent `serviceId` parameter while the repository's secondary index was defined over `serviceName`; any registration written through that path would be unreachable via `findByServiceName()`. Spring Data Redis additionally requires a no-args constructor for deserialisation, the absence of which on `Service` would cause read-side failures even after writes succeeded. These were therefore not cosmetic defects but consistency hazards: writes and reads would have referred to different fields, and the registry's view of itself would diverge from what clients observed.

The second and more interesting problem appears only when the discovery-service is replicated. The `@Scheduled` `healthCheck()` method iterates over every registered service, probes its endpoint, and increments a per-service failure counter in Redis on failure. When three replicas run this job on overlapping schedules, each performs an unsynchronised read-modify-write cycle: replica A reads `failureCount = 2`, replica B reads `2`, both compute `3`, and both write `3`. The true accumulated count is lost, and the service may be evicted earlier or later than its health warrants. This is a classical lost-update race, and because the outcome depends on which replica writes last at the master, the registry's behaviour becomes non-deterministic and observably inconsistent across clients.

## The Distributed Lock Solution

Because the health-check workload is intrinsically singleton — there is exactly one correct accumulated failure count per service per cycle — the chosen remedy is to elect a single executor per cycle rather than to coordinate concurrent executors. `ServiceRegistry.healthCheck()` now begins by attempting to acquire a Redis distributed lock via `RedisTemplate.opsForValue().setIfAbsent(key, uuid, 60s TTL)`. The `setIfAbsent` operation maps to Redis's `SET NX` and is atomic at the master, so exactly one replica observes a `true` return value per cycle. The other replicas observe `false` and return immediately, leaving health-check execution to the elected leader.

Two safety properties matter here. First, the lock has a 60-second TTL so that a replica which crashes mid-cycle cannot starve the cluster: the lock self-expires and another replica succeeds on the next tick. Second, lock release is guarded by an ownership check. Each replica writes a unique UUID as the lock value, and the `finally` block reads the current value before deleting; the delete proceeds only if the stored UUID matches. This prevents the fencing hazard in which a replica whose TTL has expired, and whose lock has been re-acquired by a peer, would otherwise blindly delete the new owner's lock. Combined with the Deployment change from `replicas: 1` to `replicas: 3`, this yields a system in which scale-out improves availability of the discovery API while leaving the singleton health-check semantics intact.

## Redis Replication and Read Consistency

The Lettuce `REPLICA_PREFERRED` policy improves read throughput by directing reads away from the master, but it introduces eventual consistency: a service registered through the master may not yet be visible on a replica chosen for the very next read. In CAP terms, the Redis master/replica deployment under network partition behaves as an AP system — replicas continue to serve potentially stale reads rather than blocking, preserving availability at the cost of strict consistency. For a discovery service this trade-off is acceptable: clients tolerate brief windows in which a freshly registered instance is not yet returned, and all writes (including the lock acquisition above) are funnelled through the master, where they remain linearisable. The tests further verify that `getAllServices` filters out `null` entries arising from expired Redis hashes whose secondary indexes have not yet been reaped.

## Test Strategy and Results

`ConsistencyTest.java` adds five Mockito unit tests that exercise the new behaviour without requiring a live Redis cluster. `only_one_replica_runs_health_checks_when_three_compete_for_lock` simulates three sequential acquisition attempts using `AtomicBoolean.compareAndSet` and asserts that `findAll()` is invoked exactly once. `lock_holder_releases_lock_after_health_check_cycle` uses `ArgumentCaptor` and `doAnswer` to capture the UUID written to Redis and replay it from `get()` during the ownership check, verifying that `delete()` is called exactly once. `replica_skips_health_checks_when_lock_not_acquired` forces `setIfAbsent` to return `false` and asserts that neither `findAll()` nor `delete()` is invoked. `registration_is_idempotent_when_same_service_registered_from_three_replicas` registers the same service three times and verifies that each call updates rather than duplicates the entry. `getAllServices_filters_null_entries_from_orphaned_redis_indexes` asserts that orphaned secondary-index entries do not leak as `null` results to API clients.

## Conclusion

Horizontal scaling exposes consistency assumptions that single-instance deployments quietly satisfy. The move to three discovery-service replicas required correcting field-mismatch bugs that would have decoupled writes from reads, introducing a Redis-backed distributed lock to serialise the health-check job with ownership-checked release, and accepting Redis's AP behaviour on the read path as a bounded trade-off. The accompanying tests pin each of these properties so that future changes cannot silently regress them. The resulting discovery-service is safe to scale horizontally while preserving a coherent view of the registry to its clients.
